## Per-provider detail

### Asana — full (confidence: high)

- **Register:** POST https://app.asana.com/api/1.0/webhooks
- **Update:** PUT https://app.asana.com/api/1.0/webhooks/{webhook_gid}
- **Deregister:** DELETE https://app.asana.com/api/1.0/webhooks/{webhook_gid}
- **Granularity:** Per-resource: a webhook is created on a specific Asana resource GID (task, project, portfolio, team, goal, workspace, section, tag, attachment, time-tracking entry). Higher-level resources (workspace, portfolio, team, goal) require mandatory filters to limit event volume. Webhooks are tied to the app/token that created them.
- **Auth:** Bearer token: either a Personal Access Token (full access) or an OAuth access token. Under Asana's scoped OAuth permissions: webhooks:write to create/update, webhooks:delete to delete (webhooks:read to list/get). Legacy full-access OAuth ("default" scope) also works.
- **Signature:** Provider-issued secret: Asana generates an X-Hook-Secret delivered during the registration handshake; every subsequent event POST carries an X-Hook-Signature header = HMAC-SHA256 of the raw request body keyed with that secret. The caller does not supply the secret. Verification is recommended but not enforced.
- **Handshake:** Yes — mandatory echo handshake. On webhook creation Asana POSTs to the target URL with an X-Hook-Secret header; the receiver must respond 200/204 and echo back the same X-Hook-Secret header while the create request is still in flight, otherwise creation fails. Localhost/invalid hostnames are rejected (403).
- **Event filtering:** Optional (mandatory for high-level resources) filters array at registration: each filter is a resource_type (+ optional resource_subtype) and action (added, changed, deleted, removed, undeleted) plus optional fields; events are delivered if they match ANY filter. PUT /webhooks/{gid} completely overwrites the filters array (target URL cannot be changed via update — recreate instead). Without filters, all events on the resource are delivered.
- **Notes:** Classification "full": arbitrary caller-chosen HTTPS callback URL, create and delete via plain REST with a PAT or OAuth token; no dashboard or app-manifest step required. Caveats for a generic ingress receiver: it must implement the synchronous X-Hook-Secret echo handshake (and store the secret per-webhook for HMAC verification), respond within the in-flight create request, and Asana sends periodic heartbeat POSTs; webhooks that repeatedly fail delivery get deactivated. Asana also caps webhooks per app per resource. Events are compact (gids + action), so the receiver typically re-fetches resources via the API.
- **Verification corrections:** All claims confirmed. Minor nuances (not errors): (1) PUT /webhooks/{webhook_gid} only allows updating the webhook's `filters` array (completely overwritten) — target URL/resource cannot be changed, so "update" support is filter-only, not full reconfiguration. (2) Registration requires a synchronous handshake: Asana POSTs X-Hook-Secret to the target URL and the server must echo it back with 200/204 before the create request returns 201 — so create at an arbitrary callback URL only works if that URL is publicly reachable and implements the handshake. (3) Docs phrase verification as "strongly recommended" (matching the claim that it is recommended but not enforced). (4) Legacy full-access OAuth uses scope=default and only works for apps registered with Full permissions; PATs act with the user's full access.
- **Sources:** https://developers.asana.com/reference/createwebhook ; https://developers.asana.com/reference/updatewebhook ; https://developers.asana.com/reference/deletewebhook ; https://developers.asana.com/docs/webhooks-guide

### Atlassian Jira Cloud — full (confidence: high)

- **Register:** POST /rest/webhooks/1.0/webhook (admin webhooks, basic auth, arbitrary URL); or POST /rest/api/3/webhook (dynamic, Connect/OAuth 2.0 apps only)
- **Update:** PUT /rest/webhooks/1.0/webhook/{webhookId} (admin webhooks). Dynamic v3 webhooks have no modify operation — only PUT /rest/api/3/webhook/refresh to extend the 30-day expiry; changes require delete + re-register.
- **Deregister:** DELETE /rest/webhooks/1.0/webhook/{webhookId} (admin webhooks); or DELETE /rest/api/3/webhook with body {"webhookIds": [...]} (dynamic)
- **Granularity:** Per Jira Cloud site (tenant/instance). Admin webhooks fire site-wide, optionally narrowed by JQL filters (project, issuetype, status, assignee, etc.). Dynamic webhooks (v3) are additionally scoped to the registering app — and for OAuth 2.0 apps, to the app+user pair that registered them (events delivered only for issues that user can see; max 5 webhooks per app per user per tenant; Connect apps max 100 per tenant).
- **Auth:** Two paths: (1) Legacy admin webhook API /rest/webhooks/1.0/webhook — basic auth (Atlassian account email + API token) of a user with Jira administrator permission; Connect apps may NOT use this API. (2) Dynamic webhook API /rest/api/3/webhook — only Connect apps or OAuth 2.0 (3LO) apps; classic scopes read:jira-work + manage:jira-webhook (granular/beta scopes: read:webhook:jira, write:webhook:jira, delete:webhook:jira). Plain API-token clients cannot use the v3 dynamic endpoints but get full CRUD via the legacy admin API.
- **Signature:** Caller-supplied HMAC secret for admin webhooks: optional `secret` field at registration (REST or admin UI); Jira sends X-Hub-Signature header as `method=signature` (HMAC-SHA256 over body, WebSub style). Dynamic webhooks for OAuth 2.0 apps instead get a JWT bearer token in the Authorization header signed with the app's client secret; Connect descriptor webhooks are signed with the app sharedSecret.
- **Handshake:** none — no echo challenge or verification token; receiver just needs a valid CA-signed HTTPS endpoint on an allowed port (443, 8080, 8443, etc.; port 80 disallowed)
- **Event filtering:** Explicit event-name list at registration (e.g. jira:issue_created, jira:issue_updated, comment_created, sprint_started...) plus optional JQL filter to restrict matching issues (jqlFilter for dynamic webhooks — required field, limited clauses: issueKey, project, issuetype, status, priority, assignee, reporter, some custom fields; `filters.issue-related-events-section` for admin webhooks). Dynamic webhooks also support fieldIdsFilter/issuePropertyKeysFilter. Sprint and version events ignore JQL/field filters. Admin webhooks support ${issue.key}-style URL variables.
- **Notes:** Classification "full" rests on the legacy admin webhook API (/rest/webhooks/1.0/webhook), which gives true create/update/delete at any HTTPS callback URL with a caller-supplied HMAC secret, no expiry — it just requires Jira-admin credentials (basic auth with API token). The newer v3 "dynamic webhooks" API is app-only with significant constraints: callback URL must use the same base URL as the app, single URL per app, 30-day expiry requiring periodic PUT /webhook/refresh (each call extends 30 days; expired webhooks recoverable for 3 months), 5-per-user limit for OAuth apps, and for non-public OAuth apps deliveries only occur when the registering user matches the app owner. For OAuth 2.0 app URLs the v3 calls go through https://api.atlassian.com/ex/jira/{cloudId}/rest/api/3/webhook. GET /rest/api/3/webhook/failed exposes delivery failures (72h window). Connect apps can also declare webhooks statically in the app descriptor (signed with sharedSecret).
- **Verification corrections:** All endpoints and core claims confirmed against official Atlassian docs. Minor refinements: (1) HMAC algorithm for admin-webhook X-Hub-Signature is not hardcoded SHA-256 — the header is `method=signature` (WebSub style) where `method` names the hash algorithm; it is currently `sha256` (docs example: `X-Hub-Signature: sha256=a4771c...`) but Atlassian explicitly warns the hash algorithm may change in the future, so verifiers should parse the method prefix. (2) Granular/beta scopes are slightly broader than claimed: POST /rest/api/3/webhook (register) requires beta scopes read:field:jira + read:project:jira + write:webhook:jira (not just write:webhook:jira); PUT /rest/api/3/webhook/refresh requires write:webhook:jira + read:webhook:jira; DELETE requires delete:webhook:jira. Classic scopes read:jira-work + manage:jira-webhook confirmed on every v3 webhook operation. The granular scopes are marked state=Beta in the OpenAPI spec. (3) "Arbitrary URL" for admin webhooks is qualified: only secure HTTPS URLs are allowed (port 80 disallowed, up to 5 redirects), and Atlassian outgoing IP ranges must be allow-listed for delivery — i.e. any external HTTPS endpoint, not literally arbitrary. (4) Register-dynamic-webhooks note: for non-public OAuth apps, webhooks are delivered only if the app owner matches the user who registered the webhook. (5) Admin permission confirmed: webhooks UI requires the "Administer Jira" global permission, and REST-registered /rest/webhooks/1.0 webhooks are explicitly called "admin webhooks" in the docs; the guide's curl examples use basic auth (--user). The guide explicitly states "Connect apps cannot register webhooks via this Jira REST API" for the legacy endpoint. (6) Confirmed there is NO modify/update operation in the v3 Webhooks group (only GET/POST/DELETE /rest/api/3/webhook, GET /rest/api/3/webhook/failed, PUT /rest/api/3/webhook/refresh), so changing a dynamic webhook does require delete + re-register; refresh extends the 30-day expiry ("Webhooks registered through the REST API expire after 30 days"). PUT /rest/webhooks/1.0/webhook/{webhookId} update confirmed, including secret semantics (omit = keep, null/empty = remove).
- **Sources:** https://developer.atlassian.com/cloud/jira/platform/webhooks/ ; https://developer.atlassian.com/cloud/jira/platform/rest/v3/api-group-webhooks/

### Auth0 — full (confidence: high)

- **Register:** POST /api/v2/log-streams (body: { name, type: "http", sink: { httpEndpoint, httpContentType, httpContentFormat: JSONLINES|JSONARRAY|JSONOBJECT, httpAuthorization, httpCustomHeaders }, filters })
- **Update:** PATCH /api/v2/log-streams/{id} (can change name, status active|paused, sink.httpEndpoint/httpAuthorization/httpContentFormat, filters, isPriority, pii_config)
- **Deregister:** DELETE /api/v2/log-streams/{id} (204 on success)
- **Granularity:** Tenant-scoped. A log stream of type "http" (Custom Webhook) delivers the tenant's log events (auth + management events) to one callback URL; up to a small per-tenant limit of streams. No per-resource (per-app/per-user) webhook scoping.
- **Auth:** Auth0 Management API v2 access token (typically obtained via client-credentials from a machine-to-machine application authorized for the Management API). Scopes: create:log_streams, read:log_streams, update:log_streams, delete:log_streams.
- **Signature:** No HMAC or public-key payload signing. Caller may supply a static secret at registration via sink.httpAuthorization, which Auth0 sends verbatim in the Authorization header of every delivery (plus optional httpCustomHeaders). Receiver must compare this shared secret itself. HTTPS endpoint required (^https://, no self-signed certs).
- **Handshake:** none — no echo challenge or verification token; the stream is created immediately. (Auth0 will move a stream to a degraded/paused state if deliveries persistently fail, but there is no registration-time handshake.)
- **Event filtering:** Optional `filters` array at registration/update; each filter is { type: "category", name: <category group> } e.g. auth.login.success, auth.login.fail, auth.signup.success, management.success, etc. Omitting filters streams ALL log events. Granularity is category-group level, not individual event-type codes. An optional startFrom cursor controls the starting log checkpoint.
- **Notes:** Auth0's modern webhook mechanism is Log Streams with the "http" (Custom Webhook) sink — full programmatic CRUD at an arbitrary HTTPS callback URL chosen by the caller, so classification is "full". Caveats: (1) it is an event/log delivery firehose of tenant log events, not per-resource webhooks; events arrive in batches per the chosen content format; (2) no cryptographic payload signature — only the caller-supplied static Authorization header value, so the generic ingress should validate that shared secret; (3) the older "Auth0 Management API Webhooks" extension and Hooks are legacy/deprecated paths and should not be used; (4) free-plan tenants are limited (historically 1 log stream; paid plans more). GET /api/v2/log-streams and GET /api/v2/log-streams/{id} exist for listing/reading (read:log_streams).
- **Verification corrections:** All endpoints and claims confirmed against official Auth0 docs. Two minor nuances: (1) PATCH status enum is active|paused|suspended (not just active|paused); 'suspended' is set by Auth0 after repeated delivery failures and can be cleared by PATCHing status to active. (2) httpContentFormat enum confirmed as JSONLINES|JSONARRAY|JSONOBJECT. Signature claim confirmed: no HMAC/payload signing exists in official docs — only the static httpAuthorization value sent verbatim in the Authorization header (plus optional httpCustomHeaders); receiver compares the shared secret itself (Auth0's own Slack integration example demonstrates this pattern). httpEndpoint must match ^https:// and self-signed certificates are not supported. Auth confirmed: Management API v2 token via client_credentials from an M2M app (audience https://{domain}/api/v2/), scopes create/read/update/delete:log_streams per endpoint. Note: a third-party guide (rollout.com) claims an 'auth0-signature' HMAC header; this is unsupported by official Auth0 documentation and should be ignored.
- **Sources:** https://auth0.com/docs/api/management/v2/log-streams/post-log-streams ; https://auth0.com/docs/api/management/v2/log-streams/patch-log-streams-by-id ; https://auth0.com/docs/api/management/v2/log-streams/delete-log-streams-by-id ; https://auth0.com/docs/customize/log-streams/custom-log-streams

### Anthropic — none (confidence: high)

- **Granularity:** Workspace-scoped (Console page lives under settings/workspaces/{workspace}/webhooks; every event payload carries organization_id and workspace_id). Webhooks only cover Managed Agents resources (session lifecycle/thread/outcome events and vault/vault_credential events) — there are no webhooks for the core Messages API, batches, usage, or billing.
- **Auth:** No API credential can manage webhook endpoints — creation, event-type selection, secret rotation, URL updates, and re-enabling after auto-disable are all done by a logged-in user in the Console UI (Manage > Webhooks). Consuming the notifications requires only the whsec_ signing secret (env var ANTHROPIC_WEBHOOK_SIGNING_KEY for the SDK unwrap helper); fetching the referenced resource afterwards uses a normal x-api-key with the managed-agents beta.
- **Signature:** Provider-issued secret: a 32-byte whsec_-prefixed signing secret is generated at endpoint creation in Console and shown only once. Each delivery carries an X-Webhook-Signature header (HMAC); SDKs verify via client.beta.webhooks.unwrap(), which also rejects payloads older than ~5 minutes. The caller cannot supply their own secret; rotation is via Console.
- **Handshake:** None — no echo challenge or verification token handshake. The receiver just needs to return any 2xx (3xx counts as failure; redirects are never followed). Constraints enforced at delivery time instead: URL must be HTTPS on port 443 with a publicly resolvable hostname; the endpoint is immediately auto-disabled if the hostname resolves to a private IP or returns a redirect, and after ~20 consecutive failed deliveries. Console can send test events, which are delivered regardless of subscription.
- **Event filtering:** Event-type list selected at registration (in Console): an endpoint subscribes to specific data.type values (e.g. session.status_idled, session.status_run_started, session.status_terminated, session.thread_created, session.outcome_evaluation_ended, vault.created/archived/deleted, vault_credential.created/archived/deleted/refresh_failed) and receives only those, plus test events.
- **Notes:** Classified "none" per the rules because configuration is dashboard-only: Anthropic's API platform has webhooks only for the Managed Agents beta (managed-agents-2026-04-01), and the docs state registration is via Console (Manage > Webhooks) with no programmatic endpoint-management API. The API reference page for Webhooks (platform.claude.com/docs/en/api/beta/webhooks) defines only inbound event payload types (BetaWebhookEvent, BetaWebhookEventData unions) and the SDK signature-verification helper (client.beta.webhooks.unwrap) — no POST/PUT/DELETE endpoints for webhook subscriptions exist anywhere in the REST reference. Payloads are thin (event type + resource IDs only); receivers must fetch the resource via GET with an API key. Retries reuse the same event.id for dedup; ordering is not guaranteed. For a Nango-style generic ingress: a customer would have to paste the Nango callback URL into the Anthropic Console manually and copy the whsec_ secret out by hand — fully automated webhook provisioning is not possible today.
- **Sources:** https://platform.claude.com/docs/en/managed-agents/webhooks.md ; https://platform.claude.com/docs/en/api/beta/webhooks ; https://platform.claude.com/settings/workspaces/default/webhooks (Console registration page referenced by docs)

### Chargebee — full (confidence: high)

- **Register:** POST https://[site].chargebee.com/api/v2/webhook_endpoints
- **Update:** POST https://[site].chargebee.com/api/v2/webhook_endpoints/{webhook-endpoint-id}
- **Deregister:** POST https://[site].chargebee.com/api/v2/webhook_endpoints/{webhook-endpoint-id}/delete
- **Granularity:** Per Chargebee site (the billing account instance, e.g. {site}.chargebee.com). Max 5 webhook endpoints per site. One endpoint can be flagged primary_url. Callback URL is caller-chosen (max 512 chars) but only ports 80, 443, 8080, 8443 are allowed.
- **Auth:** Site API key sent via HTTP Basic auth (API key as username, empty password), same as all Chargebee v2 API calls. Requires a full-access API key (read-only keys cannot create/update/delete). No OAuth scopes — Chargebee's API is API-key based, keys are created in the dashboard under Settings > Configure Chargebee > API Keys and Webhooks.
- **Signature:** No HMAC or signing secret. Optional caller-supplied HTTP Basic Auth on delivery: pass basic_auth_username and basic_auth_password (max 250 chars each) at create/update and Chargebee includes those credentials on every webhook POST. Chargebee additionally recommends HTTPS, allowlisting its published webhook source IPs (per region, listed in docs), and re-fetching events via the Events API / checking timestamp/resource_version to guard against spoofing and replays.
- **Handshake:** none — no echo challenge, verification token, or ping that the receiver must answer at registration; the endpoint just needs to return 2xx for deliveries (a manual "Test Webhook" feature exists in the dashboard but is optional)
- **Event filtering:** Explicit event-type list via repeated enabled_events[] parameter at create/update (213+ enumerated event types like subscription_created, payment_succeeded, invoice_generated). If enabled_events is omitted/empty, the endpoint receives ALL event types. List/Retrieve (GET /api/v2/webhook_endpoints[/{id}]) also available.
- **Notes:** All four CRUD operations are documented in the current official API reference (create/update/delete/list/retrieve), with curl samples and SDK support in .NET, Go, Java, Node.js, PHP, Python, Ruby; create/update/delete support idempotency keys. All writes are POST (Chargebee v2 API style; delete is POST .../delete). Other params: name (required), api_version (v1|v2 payload format), disabled, primary_url, send_card_resource, chargebee_response_schema_type (product-catalog compat). Caveat: search-engine snippets of an earlier docs revision said the webhook_endpoints API was "not enabled for live sites by default — contact support to get this enabled"; that note does NOT appear anywhere in the current (June 2026) reference pages for any of the operations (verified in the raw markdown of all regional doc mirrors), but if targeting older/live sites it may be worth a test call. Webhook payload is the full Event object; receivers should respond 2xx, and Chargebee retries failed deliveries.
- **Verification corrections:** All core claims confirmed against the official OpenAPI spec (github.com/chargebee/openapi) and official docs: POST /api/v2/webhook_endpoints (create), POST /api/v2/webhook_endpoints/{webhook-endpoint-id} (update), POST /api/v2/webhook_endpoints/{webhook-endpoint-id}/delete (delete) all exist with those exact methods/paths; no HMAC or signing secret exists — optional basic_auth_username/basic_auth_password (both maxLength 250 in the spec) at create/update; docs recommend basic auth and/or HTTPS, publish per-region webhook source IPs for allowlisting, and say to 'Check the timestamp or resource_version to guard against replay attacks' and to poll the list-events API; auth is HTTP Basic with API key as username and empty password, read-only keys are retrieve-only (writes need full-access key), no OAuth. Minor corrections: (1) dashboard path is now Settings > Configure Chargebee > 'API Keys and Events' (not 'API Keys and Webhooks'); (2) create/delete support is slightly weaker than 'arbitrary': max 5 webhook endpoints per site, url maxLength 512, and only ports 80/443/8080/8443 allowed; (3) a search snippet claiming the API is 'not enabled for live sites by default, contact support' is NOT present in the official OpenAPI spec or rendered docs for webhook_endpoints (that caveat applies only to unrelated import endpoints) and should be treated as unverified.
- **Sources:** https://apidocs.chargebee.com/docs/api/webhook_endpoints ; https://apidocs.chargebee.com/docs/api/webhook_endpoints/create-a-webhook-endpoint ; https://apidocs.chargebee.com/docs/api/webhook_endpoints/update-a-webhook-endpoint ; https://apidocs.chargebee.com/docs/api/webhook_endpoints/delete-a-webhook-endpoint ; https://apidocs.chargebee.com/docs/api/webhook_endpoints/list-webhook-endpoints ; https://apidocs.chargebee.com/docs/api/webhooks ; https://www.chargebee.com/docs/billing/2.0/site-configuration/webhook_settings

### ClickUp — full (confidence: high)

- **Register:** POST /api/v2/team/{team_id}/webhook
- **Update:** PUT /api/v2/webhook/{webhook_id}
- **Deregister:** DELETE /api/v2/webhook/{webhook_id}
- **Granularity:** Per-workspace (team_id) by default; optionally narrowed at creation to one Space, Folder, List, or Task via filter params (space_id/folder_id/list_id/task_id — the most specific location wins). Webhooks are also tied to the authorizing user's token, so deliveries reflect that user's visibility.
- **Auth:** Personal API token (pk_..., sent as `Authorization: {token}`) or OAuth2 access token (`Authorization: Bearer {token}`, authorization-code grant). ClickUp has no granular OAuth scopes — authorization is workspace-level (user picks workspaces during the OAuth flow); any valid token for the workspace can manage webhooks.
- **Signature:** Provider-issued secret: ClickUp generates `webhook.secret` and returns it in the Create Webhook response. Each delivery carries an `X-Signature` header = hex HMAC-SHA256 of the raw request body using that secret.
- **Handshake:** none — no echo challenge or verification token; the callback URL is accepted as-is at registration. Verification is purely via the X-Signature HMAC on deliveries.
- **Event filtering:** Explicit event list supplied at registration (e.g. taskCreated, taskUpdated, taskDeleted, taskStatusUpdated, list/folder/space/goal events) or `*` wildcard for all events; the list and webhook status can be changed later via the update endpoint. Plus optional location filtering (space/folder/list/task).
- **Notes:** Clean "full" case: arbitrary caller-chosen endpoint URL, full CRUD (GET /api/v2/team/{team_id}/webhook lists existing webhooks). Caveats: webhooks are bound to the authenticated user (token revocation/user removal kills the webhook), deliveries come from dynamic IPs (no IP allowlist), idempotency key is {{webhook_id}}:{{history_item_id}}, and ClickUp auto-disables unhealthy webhooks (health status on the webhook object) — receivers should 2xx quickly. No URL ownership verification, so any reachable HTTPS ingress works immediately.
- **Sources:** https://developer.clickup.com/reference/createwebhook ; https://developer.clickup.com/reference/updatewebhook ; https://developer.clickup.com/reference/deletewebhook ; https://developer.clickup.com/docs/webhooks ; https://developer.clickup.com/docs/webhooksignature ; https://developer.clickup.com/docs/authentication

### Databricks — full (confidence: high)

- **Register:** POST /api/2.0/mlflow/registry-webhooks/create
- **Update:** PATCH /api/2.0/mlflow/registry-webhooks/update
- **Deregister:** DELETE /api/2.0/mlflow/registry-webhooks/delete
- **Granularity:** Per-workspace MLflow (Workspace) Model Registry. Two scopes: model-specific webhooks (set model_name; fire for events on that registered model) or registry-wide webhooks (omit model_name; fire for events on all models in the workspace). Separately, the Notification Destinations API is workspace-scoped and destinations are then referenced per-job in jobs' webhook_notifications.
- **Auth:** Standard Databricks workspace API auth: personal access token (PAT) or OAuth (Bearer) token. No special OAuth scope; permissions are RBAC-based — CAN MANAGE on the registered model for model-specific webhooks, workspace admin for registry-wide webhooks. Notification Destinations API (the jobs-notification alternative) requires workspace admin.
- **Signature:** Caller-supplied HMAC secret: the optional `secret` field in http_url_spec at registration; Databricks sends an HMAC-SHA256 of the payload in the X-Databricks-Signature header. An optional `authorization` field can also set a static Authorization header (e.g. Bearer token) on deliveries, and enable_ssl_verification defaults to true. ### Microsoft Graph — full (confidence: high)

- **Register:** POST https://graph.microsoft.com/v1.0/subscriptions
- **Update:** PATCH https://graph.microsoft.com/v1.0/subscriptions/{id} (primarily used to renew by extending expirationDateTime; subscriptions always expire and must be renewed)
- **Deregister:** DELETE https://graph.microsoft.com/v1.0/subscriptions/{id}
- **Granularity:** Per-resource: each subscription targets a specific Graph resource path supplied in the `resource` property (e.g. me/mailFolders('Inbox')/messages, me/drive/root, chats/{id}/messages, teams/{id}/channels, users, groups, sites/{site-id}/lists/{list-id}, communications/callRecords, /communications/presences/{id}). Some resources allow tenant-wide variants (e.g. /teams/getAllMessages, chats/getAllMessages). Duplicate (same changeType + resource) subscriptions are rejected with 409.
- **Auth:** OAuth 2.0 Bearer token from Microsoft Entra ID, either delegated (work/school or personal account) or application (client-credentials) permissions depending on resource. The app needs the READ permission for the subscribed resource — e.g. Mail.Read for messages, Calendars.Read for events, Files.Read.All for OneDrive for Business, Chat.Read.All / ChannelMessage.Read.All for Teams (app-only), Group.Read.All, User.Read.All, Sites.Read.All, CallRecords.Read.All, etc. The same read scope is required to get/update/delete the subscription. Some resources are app-only (callRecords, getAll* tenant-wide feeds) or delegated-only (presence, personal OneDrive).
- **Signature:** No HMAC payload signature. Caller supplies an opaque `clientState` secret (string) at registration; Graph echoes it in every notification and the receiver must compare it to authenticate the source. For "rich notifications" that include resource data (includeResourceData=true; required pattern for chatMessage/presence/etc.), the caller must supply a public encryptionCertificate at registration — resource data is encrypted with it, and those notifications also carry Microsoft-issued JWT validationTokens the receiver can verify against Microsoft's signing keys. Delivery is HTTPS-only; optional latestSupportedTlsVersion property.
- **Handshake:** Yes — mandatory echo handshake at subscription create (and renew for lifecycleNotificationUrl). Graph sends POST {notificationUrl}?validationToken={opaqueToken} with Content-Type text/plain; the endpoint must respond within 10 seconds with HTTP 200, Content-Type text/plain, and the URL-DECODED validation token as the plain-text body. If validation fails, subscription creation returns 400 Bad Request. Additionally, every delivered notification must get a 2xx within 3 seconds or Graph retries (up to 4 hours) and may throttle/drop slow endpoints.
- **Event filtering:** Selected at registration via `changeType` (comma-separated list of created/updated/deleted, support varies by resource) combined with the specific `resource` path; a few resources accept OData $filter in the resource path (e.g. security/alerts?$filter=status eq 'New', presences?$filter=id in (...)). No global event-type catalog — one subscription = one resource + changeType set.
- **Notes:** Classification "full": create, renew (PATCH), and delete all work via the public REST API against an arbitrary caller-chosen HTTPS notificationUrl (must be publicly reachable). Key operational constraints: (1) subscriptions EXPIRE — expirationDateTime is required and max lifetime varies by resource (e.g. ~3 days for Outlook mail, longer for others); clients must renew via PATCH or re-create, and can supply a lifecycleNotificationUrl to receive reauthorizationRequired/subscriptionRemoved/missed lifecycle events. (2) Query string params on notificationUrl are preserved on delivery — useful for routing on a generic ingress. (3) Some resources mandate certificate-based payload encryption when resource data is included. (4) Graph also supports Azure Event Hubs / Event Grid as alternative delivery channels (no public endpoint needed). For a generic webhook ingress, the receiver must implement the validationToken text/plain echo and clientState comparison.

Signature and auth claims confirmed: no HMAC payload signature for notification authentication — receiver compares the echoed clientState secret; rich notifications (includeResourceData=true) require encryptionCertificate + encryptionCertificateId, encrypt resource data with the supplied public key, and include validationTokens JWTs to verify against Microsoft identity platform signing keys (expected caller appid/azp = 0bf30f3b-4a52-48df-9a82-234910c4a086); HTTPS-only delivery; latestSupportedTlsVersion is an optional property. OAuth2 Bearer from Entra ID with delegated or application permissions per resource; read scope of the subscribed resource required (Mail.Read, Calendars.Read, ChannelMessage.Read.All/Chat.Read.All app-only for Teams getAll* feeds, Group.Read.All, User.Read.All, Sites.Read.All, CallRecords.Read.All app-only, presence delegated-only Presence.Read.All, personal OneDrive delegated personal-account-only Files.Read).
